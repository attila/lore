### Open — accept as residual risk; reconsider when triggered

- **No `--draft` + `--clobber` rerun semantics** for `release.yml`. Re-running the publish job on a
  partial-state release fails fast (which is correct per the retag-fails-fast policy), but operators
  have no documented "force-fix this one upload" path. The U5 runbook's procedure (delete release +
  tag + bump version) is the authorised recovery; treat absence of a faster path as a feature.
- **Concurrent tag pushes (`v0.1.0` + `v0.1.0-hotfix.1` within seconds)** queue at the Environment
  gate in arrival order, not push order. UI shows them in reverse-chronological by gate-arrival.
  Owner can approve in the wrong order, causing `latest` to point at the hotfix briefly. Mitigation
  is operator vigilance (look at the tag name before approving). Not worth a code change without
  evidence this happens.
- **Tag with slashes (e.g. `v0.1.0/oops`)** — `tags: [v*]` admits this, the package step's `mv` then
  fails with "No such file or directory" (set -e aborts). Loud failure, wasted CI minutes only. No
  early validation of tag format in the workflow.
- **Cargo.toml profile mutation guard absent** — a future PR can flip `lto = false` / drop
  `strip = true` / loosen `opt-level`, CI passes, next release ships unoptimised binaries.
  Reviewer-discipline only today; consider a profile-snapshot test if this becomes a real concern.
- **`cross-compile` smoke job in ci.yml duplicates 4 setup steps with release.yml's `build` job.**
  Two callers below the threshold for a composite action; reconsider if a third workflow needs the
  same setup. (maintainability — recommendation: leave as-is)
- **`ci.yml` gateway hard-codes `needs[]` job list twice** — once in the `needs:` array, once in the
  bash `results=(...)`. New jobs require coordinated updates to both. Optional refactor:
  `${{ toJSON(needs) }}` + jq, or `contains(needs.*.result, 'failure')` expression. (maintainability
  — leave as-is)
- **`download-artifact` `merge-multiple: true` is silent on collision** — safe today because tarball
  names include the target, but a future addition of `lore.sig` / `manifest.json` per matrix leg
  would silently overwrite. Documentary only. (adversarial — future-fragility)
- **No retry/backoff on gh API calls** — transient 5xx surfaces as a hard failure. Manual recovery
  path is documented. Could wrap in a 3-retry loop with exponential backoff if this becomes a
  frequent paper cut. (reliability)
- **`verify` job runs `just ci` against the tagged commit** — same inherent CI threat model as
  everywhere else. `permissions: {}` + `persist-credentials: false` are the compensating controls.
  No incremental risk from this diff. (security)
- **Build matrix jobs are trusted producers of release artifacts** — any file a build job's `dist/`
  contains will be merged into publish's `dist/` and shipped (the upload glob does not enforce an
  allowlist; SHA256SUMS is computed _after_ the merge so a rogue tarball gets a sum entry too).
  Documented trust model; the owner-approval gate is the compensating control. Future tightening:
  emit only `lore-${version}-${target}.tar.gz` (one exact filename) and verify in publish before
  upload. (security)
